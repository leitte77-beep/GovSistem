A melhor abordagem é criar primeiro um assinador PAdES ICP-Brasil AD-RB para PDF, usando certificado A1 em arquivo .pfx/.p12, sem API paga, com Java + PDFBox + Bouncy Castle. Para Diário Oficial, PDF assinado é o formato mais adequado. Depois, você pode evoluir para AD-RT, AD-RC ou AD-RA, mas isso exige carimbo do tempo de uma ACT credenciada; sem uma ACT válida, o sistema não deve prometer “carimbo ICP-Brasil”.

Há um ponto importante: o A1 ainda pode ser usado, mas está em transição. O ITI informa que certificados A1 serão admitidos até 02/03/2029 e que, a partir de 03/03/2029, não será mais possível utilizar A1, A2, S1, S2, S3 e S4. Portanto, o projeto deve aceitar somente A1 agora, mas deixar a camada de chave preparada para migração futura.

1. Base normativa que o projeto deve seguir

Para assinatura ICP-Brasil, não basta “assinar com RSA”. O assinador precisa gerar a estrutura conforme os DOC-ICP de assinatura: DOC-ICP-15, DOC-ICP-15.01, DOC-ICP-15.02 e DOC-ICP-15.03. A página oficial de Documentos Principais do ITI lista esses documentos, incluindo DOC-ICP-15, DOC-ICP-15.01, DOC-ICP-15.02 e DOC-ICP-15.03.

Para o seu caso, o MVP deve usar PAdES AD-RB, porque é PDF e não exige carimbo do tempo. O DOC-ICP-15.03 define a política AD-RB baseada em PAdES e, na versão atual 1.3, o OID é 2.16.76.1.7.1.11.1.3.

A política AD-RB permite validação durante o prazo de validade dos certificados dos signatários; como não usa carimbo do tempo, validação posterior depende de referências temporais externas ou regra de negócio específica. Para AD-RT, AD-RC ou AD-RA, o DOC-ICP exige carimbo do tempo emitido por ACT credenciada na ICP-Brasil.

Em PAdES, os campos assinados mínimos incluem id-contentType, id-messageDigest, id-aa-signingCertificateV2 e id-aa-ets-sigPolicyId. O id-signingTime não deve ser usado em PAdES; o horário declarado deve ir na entrada M do dicionário PDF. O dicionário de assinatura PAdES ICP-Brasil exige, entre outros, Type, Filter, SubFilter, Contents e ByteRange, com valores padrão PBAD_PAdES e PBAD.PAdES para Filter e SubFilter.

O PDF deve ser, no mínimo, PDF 1.7; se usar PDF/A, use PDF/A-2 ou superior, porque PDF/A-1 não suporta assinaturas CAdES, requisito do PAdES ICP-Brasil. Para homologar tecnicamente seus testes, use o Validador oficial do ITI, que permite validar assinaturas eletrônicas ICP-Brasil.

Fluxo técnico da assinatura

O fluxo principal deve ser:

1. Sistema do Diário Oficial gera o PDF definitivo.
2. PDF é congelado: nenhuma alteração posterior.
3. O assinador calcula SHA-256 do PDF original.
4. O serviço carrega o certificado A1 do .pfx/.p12.
5. O serviço valida:
   - senha correta;
   - existe chave privada;
   - certificado está dentro da validade;
   - certificado é ICP-Brasil;
   - certificado é A1;
   - cadeia é confiável;
   - certificado não está revogado, quando possível.
6. O serviço prepara o PDF para assinatura incremental.
7. Cria dicionário de assinatura PAdES:
   - Type = Sig;
   - Filter = PBAD_PAdES;
   - SubFilter = PBAD.PAdES;
   - ByteRange;
   - Contents;
   - M.
8. Calcula o hash do ByteRange.
9. Gera CMS/CAdES detached com:
   - id-contentType;
   - id-messageDigest;
   - id-aa-signingCertificateV2;
   - id-aa-ets-sigPolicyId.
10. Embute o CMS no campo Contents do PDF.
11. Reabre o PDF assinado e valida a assinatura localmente.
12. Calcula SHA-256 do PDF assinado.
13. Registra auditoria.
14. Publica o PDF assinado.
15. Opcionalmente, envia o arquivo para validação manual no Validador ITI.

Para maior segurança, comece com assinatura “one-shot”: o usuário envia PFX e senha no momento da assinatura. Só implemente armazenamento de PFX se houver necessidade operacional real.

Regras de segurança obrigatórias

Nunca grave senha do PFX em texto puro. Nunca registre senha em log. Nunca registre o PFX em log. Não deixe PDF temporário ou PFX temporário em diretórios públicos. Use HTTPS obrigatório. Apague arquivos temporários após uso. Desabilite heap dump em produção, porque a senha ou chave pode aparecer na memória. Restrinja o endpoint de assinatura por autenticação forte e permissão específica.

Para PFX armazenado, use criptografia local com AES-GCM e uma chave mestra fora do banco. A senha do certificado deve ser informada no momento da assinatura ou protegida separadamente com política de rotação.
